# ADR 0004 — v2.0 Calendar contract (source, event→capacity classification, cook-night derivation, prep scheduling)

- **Status:** Accepted (design ratified 2026-07-21; v2.0 implementation deferred just-in-time — this ADR unblocks v2.0 scoping, no code lands yet)
- **Date:** 2026-07-21
- **Owner:** Backhaus
- **Relates to:** Meal Planner Bot design doc v1.0, §4 (phased delivery — resolves the dangling "classifies events by effect on cooking capacity → §10" reference), §6.1 (slot model), §6.2 (signals by mechanism), §6.3 (cook-night count), §6.5 (weather, v2.0), §10 item 4 (calendar source) + item 5 (MCP adopt-vs-build); sibling ADRs 0002 (session schema / state machine — the plan this feeds), 0003 (planner hybrid contract — `PlannerInput` this extends). **Forward-coupled to** beads `824` (day-assignment mechanics — consumes this ADR's `NightSchedule`), `bgb` (weather block on `PlannerInput`), `chj` (Todoist — materializes placed prep units), and the v3.0 revision/commit ADR (`xm4`).

---

## Context

v2.0 introduces **calendar integration**, which is what first gives the plan a notion of *specific nights* (v1.0 emits an unordered, slot-typed set; §6.1). SPEC §4's v2.0 row promises the system "classifies events by effect on cooking capacity" and points to §10 — but §10 only recorded the calendar *source* question (item 4). The bulk of v2.0 therefore had no recorded design: what a busy night means, how events map to cooking capacity, how a calendar-derived cook-night count replaces the static "3–4 weeknight + ~2 weekend" of §6.1/§6.3, and how do-ahead prep is timed. This ADR fills that gap.

The family's schedule lives across **multiple iCloud calendars** (family, per-person, kids' activities), some shared in from other iCloud accounts.

Governing principle, inherited from ADR 0003: **the code decides the hard structure; the LLM reasons over what's left.** The calendar produces a *deterministic* per-night capacity schedule; the planner (ADR 0003) then reasons over recipes against that schedule. Calendar classification is code-side and testable, not an LLM call (v1.0/v2.0); an LLM interpretation layer is a later option, not a starting requirement.

---

## Decision

### D1 — Source: read the local macOS Calendar.app via EventKit, not CalDAV or a cloud API

A small **local calendar reader** (same ownership/shape as the Apple Notes MCP, over stdio) reads events from **Calendar.app via EventKit** on the family Mac.

- **Why local, not CalDAV/Google:** the schedule spans multiple iCloud calendars including cross-account shares. Calendar.app is already the **aggregation layer** that unifies every subscribed/shared calendar the family sees; one EventKit query spans all of them. CalDAV reaches only a single iCloud account's own + inbound-shared calendars, with notoriously fiddly recurrence/timezone parsing. Google Calendar would require the schedule to actually live in Google plus a new OAuth secret. Local EventKit needs neither.
- **Consistency:** mirrors the local-first model already run for Notes — cloud read isn't needed, and there's no new 1Password secret.
- **Ops cost:** needs a **TCC "Calendars" grant on the `node` binary** (like Automation→Notes today), **re-granted after node/OS upgrades** (TCC keys on the binary). New RUNBOOK §0.1/§7 line, mirroring the existing Notes gotcha.

### D2 — Calendar selection by explicit include-list, each with a role

Config names an **allowlist** of calendars that feed cooking capacity; every unlisted calendar (US Holidays, birthdays, work-only, subscriptions) is ignored — a **denylist was rejected** (a stray new calendar would silently mark nights busy). Each included calendar carries a **role**:

- **`cook`** (e.g. "Jonathan", "Family") — an event on it removes that person as an available cook for the event's span.
- **`logistics`** (e.g. "Kids") — an event on it is assumed to **pull one cook away** (drive/attend) for its span.

The role split is what lets a kid's 4pm practice reduce capacity (a parent is driving) without a parent's 10am meeting doing so.

### D3 — Events affect a night only via the cooking window; classify FULL / QUICK / NONE

An event affects a night **only if it overlaps that night's cooking window** — config `calendar.cookingWindow` (default weekday **16:30–19:30**). A 9am appointment doesn't touch dinner; a 5–8pm game does. **All-day events cover the window.**

Per night, compute **cook availability** from the overlapping events (D2), then classify:

- **NONE** — the *entire* window is blocked for all cooks (all-day/travel event, or an event fully covering the window). Nobody home → no cook that night (leftovers/takeout).
- **QUICK** — the window is *partially* blocked, or a `logistics` run pulls a cook away for part of it. Someone's around but rushed.
- **FULL** — no capacity-reducing event overlaps.

The **QUICK "away/driving" heuristic is deterministic**: any `logistics` event overlapping the window is treated as pulling a cook. This is deliberately coarse and testable; ambiguous cases (does *this* event really mean a parent is out at dinner?) are a candidate **future LLM reasoning input**, not a v2.0 requirement.

### D4 — The `NightSchedule` is this ADR's product; it is the seam ADR-0004 ↔ bead 824

`swj` (this ADR) produces a **`NightSchedule`** — the 7 nights, each with `{date, weekday, capacity, blocking_events}`. This is a clean, independently-testable boundary:

- **This ADR owns:** "here are the nights and each one's cooking capacity, and why."
- **Bead `824` owns:** assigning the selected meals onto specific nights (mechanics, value format, validation) — it *consumes* `NightSchedule`.

Two consequences flow into the planner (ADR 0003):

- **Capacity → slot typing.** FULL weeknight = the existing constrained slot (`active_max: 60`, §6.2, unchanged). **QUICK** = a tighter gate: config `quickActiveMax` (default **30 min**) *or* a make-ahead recipe. **NONE** = no cook slot. Weekend nights stay relaxed unless an event downgrades them.
- **Slot counts are derived, not static.** ADR-0003's `PlannerInput.slots = {constrained, relaxed}` stops being static config: the orchestrator derives it from `NightSchedule` — non-NONE **weeknights** → `constrained`, non-NONE **weekend** nights → `relaxed` (QUICK further tightens that night's gate per above). `cook_night_count = #(FULL)+#(QUICK) = constrained + relaxed`. This replaces the static "3–4 weeknight + ~2 weekend" of §6.1/§6.3 (how many meals to select). Selection stays slot-typed and unordered as in v1.0; **`824` then assigns** each chosen meal to a specific dated night in the schedule. Capacity (calendar) and base slot-type (weekday) are **orthogonal axes**: weekday sets constrained-vs-relaxed, capacity can downgrade either to QUICK/NONE.

### D5 — Full prep scheduling: place prep units in v2.0, materialize as Todoist tasks in v3.0

A make-ahead recipe carries one or more **prep units**. v2.0 **places** each onto an earlier suitable night using the capacity schedule, preferring a **NONE**/leftover night, else the weekend, else the earliest FULL night before serve. Placement is an **additive, nullable** field on the plan schema (same evolution pattern as the `day` field; coordinated with bd6.13 migrations and the ADR-0002 `working_plan`).

The **phase boundary is explicit**: v2.0 does *placement and rendering only* ("prep Sun → serve Tue"). It **writes nothing**. v3.0's commit step (bead `chj` / `xm4`) turns each placed prep unit into its own Todoist task. This keeps all external writes in v3.0 while v2.0 owns the scheduling logic.

### D6 — Degrade, never fail the week, when the calendar is unavailable

No Calendar access / read failure / TCC not granted / empty schedule → **fall back to the static v1.0 count** (3–4 weeknight + 2 weekend, all treated FULL), **alert `#agent-alerts` once**, and **proceed**. This matches the degrade-don't-fail stance ratified for inline sync (bead `a9e`): a missing calendar produces a slightly-less-tailored plan, never a failed Sunday post. A `calendar.enabled` profile switch also forces this path (and keeps dev deterministic).

## Interface sketch

```ts
// D2 — config (per-context, like the rest of Config)
interface CalendarConfig {
  enabled: boolean;                          // D6 master switch (off ⇒ static fallback)
  include: { name: string; role: 'cook' | 'logistics' }[];  // D2 allowlist
  cookingWindow: { start: string; end: string };            // D3, default '16:30'–'19:30'
}
// (top-level Config also gains) quickActiveMax: number;     // D4, default 30 (minutes)

// D3/D4 — this ADR's product, consumed by bead 824
type Capacity = 'FULL' | 'QUICK' | 'NONE';

interface BlockingEvent {
  calendar: string;
  role: 'cook' | 'logistics';
  title: string;                 // raw, for the render + future LLM layer
  start: string; end: string;    // or allDay
  effect: 'removes-cook' | 'pulls-cook-away';
}
interface Night {
  date: string;                  // ISO date
  weekday: string;
  capacity: Capacity;
  blocking_events: BlockingEvent[];
}
type NightSchedule = Night[];    // 7 nights

// D4 — ADR-0003 PlannerInput changes (bgb adds the weather block separately)
interface PlannerInputV2Changes {
  slots: { constrained: number; relaxed: number };  // now DERIVED from NightSchedule,
                                                    // not static config
  cook_night_count: number;      // #(FULL)+#(QUICK) = constrained + relaxed
  // NightSchedule is carried for 824's assignment; weather (bgb) is a sibling addition
}

// D5 — additive, nullable plan-schema field (materialized to Todoist in v3.0)
interface PrepUnit {
  description: string;
  serve_date: string;            // the night this prep supports
  prep_date: string | null;      // v2.0 places it; null until placed
}
```

## Alternatives considered

- **CalDAV directly to iCloud** — rejected (D1). Single-account reach misses cross-account shared calendars the family relies on; fiddly recurrence/timezone parsing. Calendar.app already aggregates everything.
- **Google Calendar + community MCP** (§10 item 4's lead candidate) — rejected for this household. Would require the schedule to move to Google and add an OAuth secret; the local Mac already has every iCloud calendar subscribed.
- **Denylist instead of include-list** (D2) — rejected. Less config now, but a newly-subscribed calendar would silently start marking nights busy; the include-list fails safe.
- **Uniform calendar weight (no roles)** (D2) — rejected. Can't distinguish "a parent must drive" (kids' calendar) from "a parent has a daytime meeting" without roles.
- **Binary cookable/not capacity** (D3/D4) — rejected. Loses the "still cook, just fast" middle tier; a 6–8pm game shouldn't force takeout when a 25-minute meal fits.
- **Fine-grained capacity by degree of window overlap × #people** (D3) — rejected for v2.0. Most faithful, but needs thresholds tuned by observation and is the hardest to specify/test now; the coarse three-way rule is a clean start.
- **Annotate-only / suggest-prep-night without scheduling** (D5) — rejected in favor of full placement, per ratification; the phase boundary (place in v2.0, write in v3.0) keeps it from pulling task-commit concerns early.
- **LLM-driven event classification from the start** (D3) — rejected for now; the deterministic heuristic is testable and sufficient. Revisit if free-text logistics events prove genuinely ambiguous.

## Consequences

**Positive**
- **One local read covers the whole family schedule** — Calendar.app aggregates every iCloud calendar including shares; no per-account CalDAV assembly, no cloud secret.
- **Capacity is deterministic and auditable** — each night's FULL/QUICK/NONE carries its `blocking_events`, so the draft can show *why* a night is quick or off.
- **Clean seam to 824/bgb** — `NightSchedule` is a small, testable interface; day-assignment and weather layer on additively without redesign.
- **Graceful degradation** — a missing/ungranted calendar posts a static-count plan and alerts once, never fails the Sunday post (consistent with `a9e`).
- **Prep scheduling logic lands in v2.0 but writes only in v3.0** — no phase-boundary violation; placement is display-only until commit exists.

**Tradeoffs / risks**
- **Coarse logistics heuristic.** Any kids'-calendar event in the window pulls a cook, which may over-mark QUICK nights (not every practice needs a parent, or another parent stays home). Accepted as a safe-conservative default; the LLM layer (D3) is the escape hatch if it grates.
- **TCC fragility.** The Calendars grant on `node` must be re-granted after node/OS upgrades — a known, documented Notes-style gotcha, but one more thing that can silently break (mitigated by D6: an ungranted calendar degrades + alerts rather than failing).
- **New nullable plan-schema field (prep placement).** Must survive the resume path (bd6.10) and migrations (bd6.13) — the second schema evolution after `day`, and the first to carry nested placement.
- **Cooking-window and role config are household-specific.** They must be set correctly per deployment; a wrong window or a mis-tagged calendar produces subtly wrong capacity. Surfaced in the render (blocking events shown) so it's catchable by eye.

## Open items (non-blocking; resolved just-in-time before v2.0 build)

- **Concrete `include[]` calendar names + roles** and the exact `cookingWindow` — set at v2.0 deployment against the family's real calendars (config, not code).
- **`quickActiveMax` value** — default 30 min; tune during v2.0 plan-quality validation (mirrors the ADR-0003 threshold tuning).
- **NONE-night behavior in the render** — whether a NONE night is shown ("no cook — leftovers") or omitted; a presentation choice for the v2.0 render step.
- **Recurring / tentative / declined event handling** — EventKit exposes status; decide whether tentative/declined events count. Deferred to the reader implementation.
- **Interaction with weather (`bgb`)** — the per-day precipitation signal (§6.5) needs day assignment (`824`) to exist; ordering resolved there, not here.
- **Prep-unit → Todoist task shape** — owned by `chj`/v3.0 commit; this ADR only guarantees the placed `prep_date` is present to materialize.
