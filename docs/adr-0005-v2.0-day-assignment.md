# ADR 0005 — v2.0 Day assignment (populate the nullable `day`: mechanism, format, validation, render)

- **Status:** Accepted (design ratified 2026-07-21; v2.0 implementation deferred just-in-time — this ADR unblocks v2.0 scoping, no code lands yet)
- **Date:** 2026-07-21
- **Owner:** Backhaus
- **Relates to:** Meal Planner Bot design doc v1.0, §4 (phased — v2.0 "introduces day assignment"), §6.1 (slot model — "the *day mapping* is v2.0 (calendar-driven)"), §6.3 (fan-out). **Extends** ADR 0003 D3 (the `WeekPlan` output) and D5 (deterministic post-validation + one-shot repair). **Consumes** ADR 0004's `NightSchedule` (the dated nights + per-night capacity this ADR assigns onto). **Forward-coupled to** bead `bgb` (weather signals enter the same selection call), `chj` (v3.0 Todoist commit reads the assigned `day`), and the v3.0 revision ADR `xm4`. **Schema-evolution coupled to** bd6.10 (resume must survive the first `working_plan` field change) and bd6.13 (migrations).

---

## Context

v1.0 deliberately carries a **nullable `day`** through every plan/session schema (`SelectedMealSchema.day = z.null()` in `src/planner/select.ts`) so that v2.0 day assignment is *purely additive* (design §6.1, ADR 0002/0003). But the docs never specified the semantics the field was meant to enable: **who** assigns days, the **value format**, how `slot_type` constrains the day, how `validate()` extends, and how the Slack render changes. Without this, a v2.0 implementer would have to invent the core contract.

ADR 0004 now supplies the missing input: a **`NightSchedule`** — the week's dated nights, each with a `FULL | QUICK | NONE` capacity and its blocking events. Day assignment is therefore no longer "invent an ordering" but "**place the selected meals onto the specific dated nights the calendar left open**." This ADR specifies that placement.

Governing principle (inherited, ADR 0003): **the LLM decides; the code guarantees.** Assignment is a reasoning task (which meal suits which night — heavier cook on an open night, grill on a dry weekend, make-ahead on a rushed one) fenced by hard constraints the code verifies.

---

## Decision

### D1 — Assignment folds into the single selection call, not a separate pass

The existing **one** LLM selection call (ADR 0003 D1) receives the `NightSchedule` in its `PlannerInput` and emits each meal **already carrying its assigned `day`**. Selection and placement **co-reason**: the model can pick a recipe *because* of the night that's open (a make-ahead for a QUICK Tuesday; a smoker cook for a FULL, dry Saturday). Rejected alternatives (below): a second LLM assignment pass, and pure deterministic assignment — the first doubles cost and blinds selection to the nights; the second forecloses the per-day soft reasoning that weather (`bgb`) will need. Because assignment rides the existing call, **no new call and no new retry machinery** are introduced — the ADR 0003 D5 validate-and-repair loop simply gains day rules.

### D2 — `day` is an ISO date, nullable for backward/degraded compatibility

- **Format: ISO date** (`"2026-07-28"`), not a weekday name. It is the unambiguous key into `NightSchedule`; `week_key` already pins the week; the render derives the "Tuesday" label. A weekday name would be ambiguous across weeks and redundant.
- **Schema:** `SelectedMealSchema.day` changes from `z.null()` to a **nullable ISO-date string**. Nullable is load-bearing: a legacy v1.0 `working_plan` row (`day: null`) must still parse on resume, and the ADR 0004 D6 degraded path (no calendar) may leave nights unassigned. **v2.0 validation requires non-null** on a freshly generated plan (D3); the nullability is for reads, not for accepting an unassigned fresh plan.

### D3 — `validate()` gains day rules, under the same one-shot repair

`validateWeekPlan` (`src/planner/validate.ts`) extends — all deterministic, secret-free, and gated by ADR 0003 D5's single bounded repair retry:

1. **Valid night** — each `day` is a **non-NONE** night present in the `NightSchedule` (NONE nights get no meal).
2. **Slot ↔ weekday consistency** — a `constrained` meal lands on a **weeknight** date, a `relaxed` meal on a **weekend** date (the day-side counterpart of the existing pool-membership check).
3. **Capacity fit** — a meal placed on a **QUICK** night is quick-eligible: `active ≤ quickActiveMax` (ADR 0004 D4) *or* make-ahead.
4. **Distinct days** — no two meals share a date.
5. **Prep ordering** — a make-ahead meal's placed `prep_date` (ADR 0004 D5) is **strictly before** its serve `day`.

All existing v1.0 checks (exact counts, pool membership, veg-path consistency, recipe no-duplicates, untested-count) remain unchanged; the slot **counts** are now the schedule-derived ones (ADR 0004 D4). A violation re-prompts once (D5); persistent failure alerts and fails the run (ADR 0002's `catch`).

### D4 — The Slack render becomes day-ordered

`renderPlan` (`src/slack/render.ts`) changes from two `slot_type` sections ("Weeknights" / "Weekend") to **day-ordered sections, Monday→Sunday**. Each rendered night shows: the weekday label (derived from the ISO `day`), a **capacity annotation** (e.g. a "quick" / "leftovers" marker), the meal (unchanged meal rendering), and any **prep note** ("prep Sun → serve Tue", from the ADR 0004 D5 placement). Render stays a **pure function** of the enriched plan (ADR 0003 D3: selection ≠ presentation). Whether a **NONE** night is shown ("no cook — leftovers") or omitted is the render-presentation choice ADR 0004 already left open.

### D5 — First `working_plan` schema evolution; resume stays additive

`day` going `null → ISO-date` is the **first evolution of the stored `working_plan`** since v1.0. It is purely additive: the lenient resume read (`src/orchestrator/resume.ts`, bd6.10) accepts **both** a `null` day (legacy v1.0 rows, and the degraded path) **and** an ISO-date day (v2.0). An in-flight v1.0 row therefore still resumes without migration surgery; forward migrations are tracked under bd6.13. This ADR is the concrete first exercise of the "nullable-`day` carried for v2.0" bet ADR 0002 made.

## Interface sketch

```ts
// D2 — src/planner/select.ts, the only schema change
// before: day: z.null()
// after:
  day: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).nullable(),   // ISO date; null = legacy/degraded

// D1 — PlannerInput gains the schedule it assigns onto (ADR 0004 product)
interface PlannerInputDayAdditions {
  night_schedule: NightSchedule;    // ADR 0004; dated nights + capacity
  // slots {constrained, relaxed} + cook_night_count already derived from it (ADR 0004 D4)
  // weather block is a sibling addition (bgb)
}

// D3 — src/planner/validate.ts, config the day checks need
interface ValidatePlanDayConfig {
  night_schedule: NightSchedule;    // valid-night, slot↔weekday, capacity-fit checks
  quickActiveMax: number;           // ADR 0004 D4
}

// D4 — render groups by day, not slot_type (illustrative)
const meals = plan.meals
  .filter((m) => m.day !== null)
  .sort((a, b) => a.day!.localeCompare(b.day!));   // Mon→Sun via ISO date
```

## Alternatives considered

- **Two-phase: select (unordered) then a second LLM assignment pass** — rejected (D1). Isolates assignment but costs a second call + its own validate/repair, and the selector is blind to specific nights (can't pick a recipe *because* Tuesday is QUICK).
- **Deterministic code assignment after LLM selection** — rejected (D1). Cheap and fully testable, but no per-day soft reasoning: weather ("no grill if it rains Tuesday", §6.5 / `bgb`) could not steer which day a meal lands on, and day choice would be arbitrary on variety-across-days.
- **Weekday-name `day` (`"Tuesday"`)** — rejected (D2). Friendlier but ambiguous across weeks and redundant with the ISO date, which is also the `NightSchedule` key.
- **Make `day` non-nullable in v2.0** — rejected (D2). Would break resume of legacy v1.0 rows and the ADR 0004 D6 degraded path; nullability is a read-side concern, validation enforces non-null on fresh plans.
- **Amend ADR 0003 in place** — rejected in favor of a new ADR. Keeps the ratified v1.0 planner contract as a stable historical record; this ADR reads as a clean v2.0 layer (same choice as ADR 0004).

---

## Consequences

**Positive**
- **One call, unified reasoning** — selection and placement co-decide; weather layers into the same call later with no structural change.
- **Hard day rules are code-verified** — valid-night, slot↔weekday, capacity-fit, distinctness, and prep-ordering join the existing guarantees under the same one-shot repair; the draft can't silently misplace a meal.
- **Additive schema evolution proven** — the nullable-`day` bet pays off: legacy rows resume untouched, migrations are forward-only.
- **Render carries real schedule information** — day-ordered with capacity + prep notes, directly consumable by v3.0 Todoist commit (`chj` reads `day`).

**Tradeoffs / risks**
- **Larger single-call output + prompt** — the call now also emits/justifies placement and ingests the `NightSchedule`; a modest token increase (still cents; §9.3 cap unaffected) and a richer output to validate.
- **Repair burden concentrates in one loop** — a plan that violates *both* a selection rule and a day rule must be fixed in the single bounded retry; if that proves brittle, the escape hatch is to widen the repair feedback (a tuning knob, not a redesign), not a second call.
- **Render rewrite** — moving from slot-sections to day-sections is a real change to `render.ts` and its tests; the pure-function boundary (ADR 0003 D3) keeps it contained.
- **NONE-night presentation still open** — a deliberate deferral to the render implementation, not a gap in this contract.

## Open items (non-blocking; resolved just-in-time before v2.0 build)

- **NONE-night rendering** — show ("no cook — leftovers") vs omit; shared open item with ADR 0004.
- **Capacity annotation wording** in the render (how a QUICK night is marked to the family) — presentation, tuned at v2.0.
- **Repair-prompt detail for day violations** — how much placement-violation context to feed back; keep minimal to bound tokens (mirrors the ADR 0003 open item).
- **Weather → assignment** (`bgb`) — the per-day precipitation/temperature signals enter this same call; their shape and the no-grill-if-rain rule are specified there, layering on this ADR.
